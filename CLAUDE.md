# 프로젝트 규칙

## 구현/수정 요청 시 커밋·푸시 자동 포함

구현, 수정, 복구 등 코드 변경 작업을 완료하면 별도 요청 없이 커밋 및 푸시까지 한 번에 처리한다.

순서: `git fetch origin` → `git add` → `git commit` → `git push origin main` → EXE 재빌드: `pyinstaller 타겟키워드분석.spec --noconfirm`

# LLM 코딩 실수 방지 가이드라인

Tradeoff: 이 지침은 속도보다 신중함을 우선합니다. 사소한 작업은 판단에 따라 적용합니다.

## 1. 코딩 전에 생각하기

가정하지 않는다. 혼란을 숨기지 않는다. 트레이드오프를 명확히 한다.

구현 전:

- 가정을 명시적으로 서술한다. 불확실하면 질문한다.
- 여러 해석이 가능하면 모두 제시한다 — 조용히 하나만 선택하지 않는다.
- 더 단순한 접근법이 있으면 말한다. 필요하면 반대 의견을 낸다.
- 무언가 불명확하면 멈춘다. 무엇이 혼란스러운지 명시하고 질문한다.

## 2. 단순함 우선

요청한 문제를 해결하는 최소한의 코드. 추측성 코드 없음.

- 요청하지 않은 기능은 추가하지 않는다.
- 단일 사용 코드에 추상화를 넣지 않는다.
- 요청받지 않은 "유연성"이나 "설정 가능성"을 추가하지 않는다.
- 불가능한 시나리오에 대한 에러 핸들링은 하지 않는다.
- 200줄로 작성했는데 50줄로 가능하면 다시 작성한다.
- 스스로 묻는다: "시니어 엔지니어가 이게 과도하게 복잡하다고 할까?" 그렇다면 단순화한다.

## 3. 외과적 변경

꼭 필요한 부분만 건드린다. 자신이 만든 문제만 정리한다.

기존 코드를 수정할 때:

- 인접한 코드, 주석, 포맷을 "개선"하지 않는다.
- 고장나지 않은 것은 리팩토링하지 않는다.
- 다르게 하고 싶어도 기존 스타일을 맞춘다.
- 관련 없는 dead code를 발견하면 언급만 한다 — 삭제하지 않는다.

변경으로 인해 고아가 된 경우:

- 자신의 변경으로 인해 사용되지 않게 된 import/변수/함수는 제거한다.
- 기존의 dead code는 요청받지 않으면 제거하지 않는다.

테스트: 변경된 모든 줄은 사용자의 요청으로 직접 추적 가능해야 한다.

## 4. 목표 기반 실행

성공 기준을 정의한다. 검증될 때까지 반복한다.

작업을 검증 가능한 목표로 변환한다:

- "유효성 검사 추가" → "잘못된 입력에 대한 테스트 작성 후 통과시키기"
- "버그 수정" → "버그를 재현하는 테스트 작성 후 통과시키기"
- "X 리팩토링" → "리팩토링 전후 테스트 통과 확인"

다단계 작업의 경우 간략한 계획 서술:

1. [단계] → 검증: [확인]
2. [단계] → 검증: [확인]
3. [단계] → 검증: [확인]
